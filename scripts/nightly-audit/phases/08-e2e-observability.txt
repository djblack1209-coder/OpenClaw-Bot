【本阶段任务：端到端业务流程与可观测性审计】

A. Telegram Bot 业务逻辑完整性（逐 Bot 检查）
- 枚举所有 7 个 Bot 的命令处理器和回调处理器
- 检查每个命令是否有: 权限校验 → 参数解析 → 业务执行 → 结果回复 → 错误回复 的完整流程
- 检查命令之间是否有冲突（同一个命令被多个 Bot 注册）
- 检查回调按钮点击后是否有响应（不能点了没反应）
- 检查长耗时操作是否有"正在处理"的提示（typing action 或 loading 消息）
- 检查消息路由是否正确（群聊消息 vs 私聊消息的处理差异）

B. 消息处理链路完整性
- 追踪一条用户消息的完整生命周期:
  用户输入 → Telegram API 接收 → Bot 路由 → NLP/AI 处理 → 回复生成 → 发送回复
- 检查每个环节是否有错误处理和降级策略
- 检查消息处理的超时保护（Telegram 对回复有 30s 限制）
- 检查是否有消息去重机制（网络抖动导致重复消息）
- 检查大消息（超过 4096 字符）的分割处理

C. 定时任务端到端验证
- 枚举所有 APScheduler 任务（social_scheduler.py + execution/scheduler.py + trading schedulers）
- 检查每个定时任务的: 触发条件 → 数据获取 → 业务处理 → 结果通知 链路
- 检查定时任务失败时是否有告警（不能静默失败）
- 检查定时任务的日志是否可追溯（知道哪次执行成功/失败）
- 验证 cron/launchd 任务与 APScheduler 任务之间没有资源冲突

D. 日志结构化与可搜索性
- 检查日志是否使用结构化格式（推荐 JSON 格式）
- 检查是否有统一的 trace_id / request_id 贯穿整个请求链路
- 检查日志级别使用是否合理:
  - ERROR: 需要人工介入的问题
  - WARNING: 异常但可自愈的情况
  - INFO: 关键业务事件
  - DEBUG: 开发调试信息（生产环境不应输出）
- 检查生产环境日志级别是否为 INFO（不是 DEBUG）
- 检查日志中是否有足够的上下文信息（user_id, bot_name, command 等）

E. 健康检查端点审计
- 检查 FastAPI 是否有 /health 端点
- 检查 /health 返回的信息是否全面:
  - 应用版本、启动时间、运行时长
  - 数据库连接状态
  - Redis 连接状态
  - 各 Bot 实例在线状态
  - 最近一次定时任务执行时间
  - 内存/CPU 使用情况（如可获取）
- 如果没有健康检查端点，直接创建一个
- 检查 Docker 健康检查是否调用了 /health 端点

F. 告警与通知机制
- 检查哪些异常情况会触发告警（应该覆盖: 服务崩溃、API调用连续失败、数据库连接断开、磁盘空间不足）
- 检查告警渠道是否可用（Telegram 通知是否配置正确）
- 检查是否有告警去重/抑制（避免同一个问题重复发送上百条通知）
- 检查是否有严重告警的升级机制（小问题通知群、大问题通知个人）

G. 进程管理与自愈
- 检查 launchd plist 文件中的重启策略（KeepAlive, ThrottleInterval）
- 检查 Docker 容器的 restart policy
- 检查进程崩溃后的恢复时间（冷启动需要多久）
- 检查是否有启动顺序依赖（Redis 必须先于应用启动）
- 检查 graceful shutdown 是否正确实现（收到 SIGTERM 后完成当前请求再退出）

H. 网络与基础设施
- 检查所有出站网络请求的超时配置（不能无限等待）
- 检查 DNS 解析是否有缓存（避免频繁 DNS 查询）
- 检查服务器防火墙规则（如可获取配置文件）
- 检查 SSH 配置安全性（禁用密码登录、只允许密钥）
- 检查 SSL 证书有效期（如有 HTTPS 配置）

I. 性能基线与异常检测
- 检查是否有请求耗时记录（中间件或装饰器）
- 检查是否有慢查询日志
- 检查是否有内存泄漏的征兆（长期运行后内存持续增长的代码模式）
- 检查是否有定期重启机制（作为内存泄漏的兜底策略）
