A. 测试健康度

- 分析失败的测试，逐个修复

B. 语法与类型检查
- 运行 ruff 检查: ruff check packages/clawbot/src/ --select E,F,W

- 检查所有 import 是否可解析（无 ModuleNotFoundError 风险）
- 修复所有 F 级别（Fatal）错误

C. 错误处理审计
- 检查所有外部 API 调用是否有 try/except
- 检查 except 块是否有意义（不是空 pass 忽略关键错误）
- 检查 httpx/aiohttp 调用是否有超时设置
- 检查数据库操作是否有事务保护
- 检查文件操作是否有 with 语句或显式 close

D. 并发安全审计
- 检查 asyncio 代码中是否有竞态条件
- 检查共享状态（全局字典、类属性）是否有锁保护
- 检查 asyncio.Lock 使用是否正确（不能在 __init__ 中创建 event loop 相关对象）
- 检查数据库连接池配置

E. 资源泄漏审计
- 检查 HTTP 客户端是否正确关闭（async with 或显式 close）
- 检查文件句柄是否正确关闭
- 检查 WebSocket 连接是否有 close 处理
- 检查定时任务是否在关机时正确取消

F. 功能完整性检查（重点！）
- 检查是否有只实现了一半的功能（函数体只有 pass 或 raise NotImplementedError）→ 直接补全实现
- 检查是否有被注释掉的功能代码 → 分析意图，要么恢复并完善，要么彻底删除
- 检查是否有 TODO/FIXME 标记的未完成工作 → 直接完成它
- 如果发现未完成的功能，先搜索 GitHub 是否有成熟的开源方案可以搬运集成
- 搬运优先：成熟开源方案 > 改造适配 > 从零手写
- 不要留下任何半成品，要么完成要么删除

G. 性能隐患
- 检查是否有 N+1 查询模式
- 检查是否有不必要的同步阻塞调用（sync HTTP in async context）
- 检查大循环中是否有可优化的低效操作
- 检查内存使用（大列表、不必要的数据复制）

H. 代码重复度分析
- 检查是否有多个文件中存在高度相似的代码块
- 特别关注: 错误处理模板、API 调用模式、数据格式化逻辑
- 如发现重复代码 → 提取为公共函数/Mixin
- 对比相似模块之间是否有可合并的逻辑

I. 多 Bot 实例并发安全
- 本项目运行 7 个 Bot 实例，检查共享资源竞争风险
- 检查全局配置/状态是否在 Bot 实例间正确隔离
- 检查 API 配额是否在多个 Bot 之间合理分配
- 检查数据库/Redis 操作是否存在跨实例的行锁争抢
