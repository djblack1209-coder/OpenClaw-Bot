【本阶段任务：架构与运维审计】

A. 代码架构审计
- 检查模块间依赖是否合理（是否有循环依赖）
- 检查分层是否清晰: src/bot/ (表现层) → src/core/ (业务层) → src/执行层
- 检查是否有"上帝类"（单个类/文件超过 800 行未拆分）
- 检查接口设计是否遵循单一职责原则
- 对比 MODULE_REGISTRY.md 与实际模块，补全缺失的注册

B. 依赖管理
- 检查 requirements.txt 与实际 import 是否一致（有无多余/缺失）
- 检查 package.json 与实际使用是否一致
- 检查 Cargo.toml 依赖版本是否合理
- 检查是否有可合并的相似功能依赖（如多个 HTTP 库）
- 确认 docs/registries/DEPENDENCY_MAP.md 与实际依赖一致

C. CI/CD 审计
- 审查 .github/workflows/ci.yml 是否覆盖足够
- 检查 CI 是否运行了: 单元测试 + 类型检查 + lint + 安全扫描
- 检查 CI 缓存策略是否有效
- 检查分支保护规则（main 是否需要 CI 通过才能合并）
- 如有改进空间，直接修改 CI 配置

D. Docker 与部署
- 审查 docker-compose.yml 配置
  - 服务依赖关系是否正确
  - 资源限制是否合理（2核2G服务器）
  - 健康检查是否配置
  - 重启策略是否合理
- 审查 docker-compose.newapi.yml
- 检查是否需要 .dockerignore 文件
- 检查部署文档 docs/guides/DEPLOYMENT_GUIDE.md 是否与实际配置一致

E. 日志与监控
- 检查日志级别配置是否合理（生产环境应为 INFO，不是 DEBUG）
- 检查日志轮转配置（参考 scripts/setup_log_rotation.sh）
- 检查是否有足够的运行时指标收集
- 检查健康检查端点是否存在且有效

F. 错误恢复与容灾
- 检查进程崩溃后是否能自动重启
- 检查数据库/Redis 连接断开后的重连机制
- 检查 Telegram Bot 断线重连
- 检查文件系统空间不足时的处理
- 检查 docs/guides/DISASTER_RECOVERY.md 是否完整可用

G. 性能与可扩展性
- 检查异步代码是否正确使用了 async/await（没有在 async 中做同步阻塞）
- 检查数据库查询是否有索引
- 检查是否有适当的缓存层
- 检查定时任务调度是否合理（不会同时触发大量任务）

H. macOS 桌面入口
- 检查 Tauri 应用是否有 .app 产物或构建命令
- 检查 apps/ 目录下是否有可用的桌面入口
- 检查 Makefile 中是否有桌面端构建目标
- 如有构建脚本，验证其可用性

I. 实际构建验证
- 运行 cd apps/openclaw-manager-src && npm run build（验证前端构建产物）
- 运行 cd apps/openclaw-manager-src/src-tauri && cargo build（验证 Rust 编译）
- 检查构建产物目录是否正确配置在 .gitignore 中
- 如构建失败，分析原因并修复

J. 版本号管理
- 检查 package.json / Cargo.toml / pyproject.toml 中的版本号是否一致
- 检查 tauri.conf.json 中的应用版本号
- 检查是否遵循语义化版本规范（SemVer）
- 检查 Git tag 是否与版本号对应

K. 定时任务专项审计
- 枚举所有 APScheduler/cron 定时任务
- 检查执行频率是否合理（不会同时触发大量任务）
- 检查是否有重叠防护（上一次未完成时下一次不启动）
- 检查失败时是否有通知/重试机制
