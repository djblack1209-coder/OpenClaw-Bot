【本阶段任务：数据库与交易系统专项审计】

A. 数据库 Schema 完整性
- 检查所有 SQLite 数据库文件的表结构是否有主键
- 检查是否有缺失的索引（高频查询字段必须有索引）
- 检查字段类型是否合理（金额用 DECIMAL 而非 FLOAT）
- 检查是否有孤立的外键或悬空引用
- 检查 ORM 模型（如有）与实际表结构是否一致
- 检查是否有数据库迁移机制（Alembic/手动 DDL）

B. 数据一致性审计
- 检查事务边界是否正确（关键操作是否在同一个事务内）
- 检查并发写入是否有锁保护（SQLite WAL 模式是否启用）
- 检查是否有"先读后写"导致的 TOCTOU 竞态
- 检查数据校验约束（NOT NULL、CHECK、UNIQUE）是否充分
- 检查 Redis 缓存与数据库的一致性策略（缓存失效/更新时机）

C. 数据备份与恢复
- 检查是否有自动备份机制（定时任务 scheduler.py 中的 daily DB backup）
- 验证备份脚本是否真正可用（能不能恢复）
- 检查备份存储位置是否安全（不在数据库同目录）
- 检查备份保留策略（保留多少天、是否自动清理旧备份）
- 检查是否有备份完整性校验（校验和/大小检查）

D. 交易系统风控审计
- 检查 packages/clawbot/src/trading/ 下所有模块的风控逻辑
- 检查单笔交易金额上限是否存在且合理
- 检查每日交易总额上限是否存在
- 检查止损/止盈逻辑是否正确实现
- 检查异常波动下是否有熔断机制
- 检查交易信号的阈值参数是否有合理的默认值和范围校验

E. 交易订单状态机
- 检查订单状态流转是否完整（创建→确认→执行→完成/取消）
- 检查是否有不可达状态或死锁状态
- 检查状态回退是否有保护（已执行的订单不能回到创建状态）
- 检查异常中断时的订单状态恢复机制
- 检查是否有订单超时处理（挂起太久的订单）

F. 资金安全
- 检查余额计算是否使用 Decimal 类型（避免浮点精度问题）
- 检查是否有负余额保护
- 检查转账/扣款操作是否有双重确认机制
- 检查是否有交易流水记录（可追溯、可对账）
- 检查模拟交易与真实交易是否严格隔离

G. 交易日志与审计追踪
- 检查 data/audit/operations.jsonl 是否正确记录交易操作
- 检查审计日志是否包含: 时间戳、操作者、操作类型、金额、结果
- 检查审计日志是否不可篡改（append-only）
- 检查是否有异常交易告警机制
- 检查 /omega/audit API 端点返回的数据是否完整

H. 定时交易任务
- 检查 _scheduler_daily.py 和 _scheduler_tasks.py 中的定时任务
- 检查定时任务执行失败时的重试和通知机制
- 检查定时任务是否有重叠防护（上一次没跑完下一次不启动）
- 检查市场休市时间是否正确处理（周末/节假日不交易）

【修复规范】
- 金额计算必须使用 decimal.Decimal，禁止 float
- 数据库操作必须有事务保护
- 交易操作必须有完整的日志记录
- 风控参数必须可配置（环境变量或配置文件），不硬编码
