# OpenClaw Bot 安全政策

> 类型: 架构/设计
> 最后更新: 2026-06-22
> 目标: 明确漏洞报告方式、密钥处理规则和项目安全边界。

## 一、支持范围

当前 `main` 分支是主要维护分支。安全修复会优先进入 `main`，再由维护者决定是否回补到发布标签或部署分支。

## 二、如何报告漏洞

如果你发现安全问题，请优先使用 GitHub 的私密安全报告入口：

1. 打开仓库 `Security` 页面。
2. 选择 `Report a vulnerability`。
3. 写清影响范围、复现步骤和建议修复方向。

如果 GitHub 私密安全报告不可用，请先开一个不包含敏感细节的 issue，标题写“Security contact needed”，等待维护者提供安全沟通方式。

请不要在公开 issue、PR、评论或截图中发布：

- API Key、token、Cookie、Session、OAuth code
- Telegram Bot token 或 chat id 组合
- 交易账户、券商凭证、支付凭证
- 浏览器 Profile、二维码、验证码、短信内容
- 内网地址、生产数据库、用户隐私数据

## 四、项目安全边界

OpenClaw Bot 包含 Bot、浏览器自动化、配置助手、交易分析、社媒和电商客服等模块。贡献和使用时必须遵守以下边界：

1. **密钥保护**：真实 Key 只放在本机 `.env` 或部署平台密钥管理中，不能提交到仓库。
2. **低敏输出**：日志、测试、截图和 API 响应只能展示状态、数量、Provider 名称或脱敏后缀。
3. **人工确认**：交易、支付、账号授权、服务停止、外部发布等高风险动作必须保留人工确认。
4. **平台合规**：不得新增绕过验证码、登录保护、付费墙、风控或服务条款的逻辑。
5. **最小权限**：新增 API、Webhook、后台任务和文件读写默认使用白名单、路径限制和鉴权。
6. **安全失败**：外部服务不可用、Key 失效、权限不足时，系统应降级或提示，不应泄露敏感上下文。

## 五、AI 模型和 API credits 使用安全要求

如果使用 OpenAI API credits 或其他赞助资源维护本项目，只允许用于开源维护工作：

- 代码审查
- 测试生成
- 文档整理
- 安全分析
- 重构建议

禁止用于：

- 真实交易决策或自动下单
- 规避平台风控、验证码或访问限制
- 批量刷量、垃圾信息、欺骗性社媒操作
- 未授权数据抓取
- 商业客户工作负载或 credits 转售

## 六、安全检查清单

提交 PR 前请确认：

- [ ] 没有新增或提交 `.env`、Key、Cookie、token、数据库、浏览器 Profile。
- [ ] 没有在日志和测试断言里输出敏感明文。
- [ ] 外部输入已做校验或白名单限制。
- [ ] 高风险命令需要确认码或人工审批。
- [ ] 新增网络调用有超时、异常处理和低敏错误提示。
- [ ] 文档已说明风险边界和验证方式。

## 七、响应预期

维护者会尽量按下面优先级处理：

| 严重度 | 示例 | 目标响应 |
|--------|------|----------|
| 高 | 密钥泄露、未授权执行命令、绕过高风险确认 | 尽快确认并修复 |
| 中 | 鉴权缺口、敏感日志、路径越权 | 优先排期 |
| 低 | 文档不清、低敏提示不足、非默认危险配置 | 常规维护 |

实际响应时间取决于维护者可用时间和复现完整度。
